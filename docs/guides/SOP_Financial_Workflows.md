# Standard Operating Procedure: Financial Transaction Management

**Document ID:** SOP-FIN-001
**Version:** 1.0
**Date:** 29-Jan-2026
**Department:** Finance & Accounts
**System:** Gyan Manjari Portal

---

## 1. Overview
This document outlines the mandatory procedures for handling **Cheque Returns (Bounces)** and **Fee Refunds** within the Gyan Manjari Portal. All finance officers and administrators must adhere to these workflows to ensure accurate financial reporting and balance tracking.

---

## 2. Cheque Bounce Process (Dishonored Payments)

### 2.1 Scenario
A student submits a fee payment via Cheque. The receipt is generated, but the bank later returns the cheque unpaid due to insufficient funds, signature mismatch, or other reasons.

### 2.2 Functional Impact
- The student's fee balance must reflect the unpaid amount.
- The original receipt must be invalidated.
- The system must track the failed transaction.

### 2.3 Procedure
**Role Required:** Accountant / Principal / Super Admin

1.  **Locate the Transaction**:
    - Go to **Financial Reports** > **Payment History**.
    - Search for the student using **Name** or **Admission Number**.
    - Scroll to the **Payment Timeline** section.

2.  **Cancel the Receipt**:
    - Identify the specific transaction associated with the bounced cheque number.
    - Click the **Cancel Receipt** button (Red icon).
    - **Mandatory**: In the popup modal, enter a detailed reason.
        - *Format Recommendation*: `[Bank Reason] - Chq No: [Number] - Date: [Date]`
        - *Example*: `Insufficient Funds - Chq No: 123456 - Date: 25-Jan-2026`
    - Click **Yes, Cancel it!**.

3.  **System Verification**:
    - Verify that the receipt status is now tagged as **CANCELLED**.
    - Verify that the **Total Pending** amount for the student has increased by the cheque amount.

4.  **Penalty Application (Offline Step)**:
    - If the institution levies a Cheque Bounce Penalty, manually add a "Miscellaneous Fee" or "Fine" to the student's profile (depending on current system configuration).

## 3. Fee Refund Process

### 3.1 Scenario
Refunds are processed for excess payments, double payments, or admission withdrawals.

### 3.2 Supported Modes
- **Online Refund**: For payments made via Payment Gateway (EaseBuzz). Refund is credited back to the source account.
- **Manual Refund**: For Cash/Cheque payments. The system tracks the record, but the actual payout is done manually.

### 3.3 Procedure
**Role Required:** Admin (Approval) / Finance (Processing)

#### Phase 1: Initiation
1.  Navigate to **Fees** > **Refund Management**.
2.  Requests may be auto-generated by students (Student Portal) or raised manually by Admin.

#### Phase 2: Review & Approval
1.  Locate the request in the **Pending** tab.
2.  Click **View Details** (Eye Icon) to inspect the original payment and reason.
3.  **Action**:
    - **Mark Under Review**: If investigation is needed.
    - **Approve**: If valid. Add remarks if necessary.
    - **Reject**: If invalid. **Note:** A rejection reason is mandatory and will be visible to the student.

#### Phase 3: Disbursement (Processing)
1.  Go to the **Approved** tab.
2.  Click **Process Refund**.
3.  **Confirmation**:
    - The system will communicate with the Payment Gateway (for validity checks).
    - Upon success, the Status updates to **Refunded**.
    - The Payment Record in `tbl_payments` is tagged as `Refunded`.
    - The system automatically clears the "Token Fee" flag if the refund is for a Token Fee payment.

---

## 4. Important Notes
- **Do NOT** delete receipts from the database directly. Always use the **Cancel** function to maintain an audit trail.
- Refunds for Online Payments usually take **5-7 business days** to reflect in the payee's bank account. Be sure to communicate this expectation.
- For technical support regarding failed refunds or stuck transactions, contact the System Administrator immediately with the **Transaction ID**.
